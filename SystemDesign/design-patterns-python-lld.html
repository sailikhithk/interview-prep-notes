What a Design Pattern Is (and Isn't)

A design pattern is a named, reusable solution to a recurring object-design problem. It is not a library or a copy-paste snippet — it is a shape of collaboration between classes that you adapt to your context. In LLD interviews, naming the pattern signals you recognize the underlying force (coupling, extension, object creation cost) the problem is really about.

Creational

Control how objects are made, decoupling clients from concrete classes.

Structural

Compose objects into larger structures while keeping them flexible.

Behavioral

Assign responsibilities and communication between objects.

Python caveat: Python's first-class functions, duck typing, modules, and decorators make several GoF patterns nearly invisible. Each card below shows the textbook OO form and the idiomatic Python shortcut, so you know when the full ceremony is worth it.

How to Pick a Pattern (Decision Guide) Cheat

"Part-whole hierarchies / many tiny shared objects" → Composite, Flyweight.

CREATIONAL · 5

Creational Patterns

How objects get created — decoupling clients from concrete construction.

1. Singleton Creational

Intent: guarantee a class has exactly one instance and provide a global access point to it.

When (LLD): shared config, a connection/thread pool, a logger, an in-memory cache — where multiple instances would be wasteful or inconsistent.

class Singleton:
    _instance = None
    def __new__(cls, *args, **kwargs):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance

class Config(Singleton):
    def __init__(self):
        self.settings = {"env": "prod"}

a, b = Config(), Config()
assert a is b   # same object

Pythonic idiom: just use a module. A Python module is imported once and cached in sys.modules — module-level objects are de-facto singletons. Or use @functools.lru_cache / a @cache-decorated factory function.

Gotcha: singletons are global state — they hurt testability and hide dependencies. Not thread-safe by default (guard __new__ with a lock). Prefer dependency injection where you can.

2. Factory Method Creational

Intent: define an interface for creating an object, but let subclasses (or a method) decide which concrete class to instantiate.

When (LLD): the client shouldn't hard-code PayPalGateway() vs StripeGateway(); it asks a factory and stays decoupled from concrete types.

from abc import ABC, abstractmethod

class Notifier(ABC):
    @abstractmethod
    def send(self, msg): ...

class Email(Notifier):
    def send(self, msg): print("email:", msg)
class SMS(Notifier):
    def send(self, msg): print("sms:", msg)

def notifier_factory(kind: str) -> Notifier:
    return {"email": Email, "sms": SMS}[kind]()

notifier_factory("sms").send("hi")

Pythonic idiom: a dict dispatch ({"email": Email}) or a registry decorator replaces a class hierarchy of factories. Classes are first-class, so you store the class itself, not a factory subclass.

Gotcha: don't confuse with Abstract Factory — Factory Method makes one product via subclassing/override; Abstract Factory makes families.

3. Abstract Factory Creational

Intent: provide an interface to create families of related objects without specifying their concrete classes.

When (LLD): cross-platform UI kits (Win vs Mac buttons + checkboxes), or DB driver families where products must be used together consistently.

class Button: ...
class Checkbox: ...
class MacButton(Button): pass
class MacCheckbox(Checkbox): pass
class WinButton(Button): pass
class WinCheckbox(Checkbox): pass

class GUIFactory(ABC):
    @abstractmethod
    def button(self): ...
    @abstractmethod
    def checkbox(self): ...

class MacFactory(GUIFactory):
    def button(self): return MacButton()
    def checkbox(self): return MacCheckbox()

def build_ui(factory: GUIFactory):
    return factory.button(), factory.checkbox()

Pythonic idiom: group the family in a small class or namespace and pass it around; or use a dict of dicts keyed by platform → product type.

Gotcha: adding a new product type means changing every factory — the interface is closed to new products but open to new families.

4. Builder Creational

Intent: separate the construction of a complex object from its representation, so the same steps can build different results.

When (LLD): objects with many optional fields (HTTP request, SQL query, a Pizza), avoiding telescoping constructors with 10 positional args.

class Burger:
    def __init__(self): self.parts = []

class BurgerBuilder:
    def __init__(self): self._b = Burger()
    def add(self, part):
        self._b.parts.append(part)
        return self          # fluent chaining
    def build(self): return self._b

burger = (BurgerBuilder()
          .add("bun").add("patty").add("cheese")
          .build())

Pythonic idiom: keyword arguments with defaults, @dataclass, or **kwargs usually remove the need. Reach for Builder only when construction has ordering rules or validation steps.

Gotcha: a fluent builder that returns self is convenient but makes partially-built objects observable — only expose the product via build().

5. Prototype Creational

Intent: create new objects by cloning an existing instance (the prototype) rather than constructing from scratch.

When (LLD): object creation is expensive (heavy DB load, deep config) and you need many near-identical copies; or you want to snapshot a configured instance as a template.

import copy

class Document:
    def __init__(self, styles, content):
        self.styles, self.content = styles, content
    def clone(self):
        return copy.deepcopy(self)

template = Document({"font": "Inter"}, [])
report = template.clone()        # cheap, independent copy
report.content.append("Q3 numbers")

Pythonic idiom: the pattern is the stdlib — copy.copy (shallow) and copy.deepcopy (deep). Customize via __copy__ / __deepcopy__.

Gotcha: shallow vs deep copy bugs — a shallow clone shares nested mutable objects, so mutating the copy can corrupt the prototype.

STRUCTURAL · 7

Structural Patterns

How objects and classes compose into larger, flexible structures.

6. Adapter Structural

Intent: convert the interface of a class into another interface clients expect, letting incompatible classes work together.

When (LLD): wrapping a third-party SDK or legacy class so it satisfies your interface without touching its source.

class LegacyPrinter:                 # adaptee (can't change)
    def print_text(self, t): print("LEGACY:", t)

class Printer(ABC):                  # target interface
    @abstractmethod
    def out(self, text): ...

class PrinterAdapter(Printer):
    def __init__(self, legacy): self._legacy = legacy
    def out(self, text):             # translate call
        self._legacy.print_text(text)

PrinterAdapter(LegacyPrinter()).out("hello")

Pythonic idiom: duck typing means you often just add a method or use a thin wrapper function. functools.partial or a small closure can adapt a callable signature.

Gotcha: Adapter changes an interface (after-the-fact fix); Bridge designs two axes up front; Decorator adds behavior with the same interface. Know the difference.

7. Bridge Structural

Intent: decouple an abstraction from its implementation so the two can vary independently.

When (LLD): you'd otherwise get a class explosion (Shape × RenderAPI = RasterCircle, VectorCircle, RasterSquare...). Bridge turns multiplication into addition.

class Renderer(ABC):                 # implementation axis
    @abstractmethod
    def draw_circle(self, r): ...
class Vector(Renderer):
    def draw_circle(self, r): print("vector circle", r)
class Raster(Renderer):
    def draw_circle(self, r): print("pixels circle", r)

class Shape:                          # abstraction axis
    def __init__(self, renderer): self.renderer = renderer
class Circle(Shape):
    def __init__(self, renderer, r):
        super().__init__(renderer); self.r = r
    def draw(self): self.renderer.draw_circle(self.r)

Circle(Vector(), 5).draw()

Pythonic idiom: plain composition — inject the implementation object in __init__. Python rarely needs the formal abstraction/implementor class trees.

Gotcha: Bridge is a design-time decision (two dimensions you know will both grow); Adapter is a retrofit. Same composition shape, different intent.

8. Composite Structural

Intent: compose objects into tree structures and let clients treat individual objects and compositions uniformly.

When (LLD): file systems (file vs folder), UI widget trees, org charts, nested menus — any part-whole hierarchy.

class Node(ABC):
    @abstractmethod
    def size(self) -> int: ...

class File(Node):                    # leaf
    def __init__(self, n): self.n = n
    def size(self): return self.n

class Folder(Node):                  # composite
    def __init__(self): self.children = []
    def add(self, c): self.children.append(c)
    def size(self):
        return sum(c.size() for c in self.children)

root = Folder(); root.add(File(10)); root.add(File(20))
print(root.size())   # 30 — same .size() call on leaf or tree

Pythonic idiom: recursion over nested lists/dicts often suffices; the value of formal Composite is the uniform interface across leaf and container.

Gotcha: leaves don't support add()/remove() — either raise on leaves or keep child-ops only on the composite (the safer, type-honest choice).

9. Decorator Structural

Intent: attach additional responsibilities to an object dynamically by wrapping it — a flexible alternative to subclassing.

When (LLD): stackable add-ons — coffee + milk + sugar, or an I/O stream + buffering + compression + encryption, each layered independently.

class Coffee:
    def cost(self): return 2.0

class MilkDecorator:
    def __init__(self, wrapped): self.wrapped = wrapped
    def cost(self): return self.wrapped.cost() + 0.5

class SugarDecorator:
    def __init__(self, wrapped): self.wrapped = wrapped
    def cost(self): return self.wrapped.cost() + 0.2

drink = SugarDecorator(MilkDecorator(Coffee()))
print(drink.cost())   # 2.7

Pythonic idiom: Python's @decorator syntax is exactly this pattern for functions (wrap behavior). For objects, composition as above; functools.wraps preserves metadata.

Gotcha: decorators must honor the wrapped object's interface (delegate unknown calls via __getattr__), or callers break when they reach past the methods you overrode.

10. Facade Structural

Intent: provide a single, simplified interface over a complex subsystem of many classes.

When (LLD): an OrderService.checkout() that hides inventory, payment, shipping, and email subsystems behind one call.

class Inventory:
    def reserve(self, item): print("reserved", item)
class Payment:
    def charge(self, amt): print("charged", amt)
class Shipping:
    def dispatch(self, item): print("shipped", item)

class OrderFacade:                    # one simple door
    def __init__(self):
        self.inv, self.pay, self.ship = Inventory(), Payment(), Shipping()
    def checkout(self, item, amt):
        self.inv.reserve(item)
        self.pay.charge(amt)
        self.ship.dispatch(item)

OrderFacade().checkout("book", 25)

Pythonic idiom: a module with a few top-level functions is a natural facade over its internal helpers — the public API is the facade.

Gotcha: a facade should simplify, not become a god-object. Keep subsystems still usable directly for advanced needs — don't lock everything behind it.

11. Flyweight Structural

Intent: minimize memory by sharing the common (intrinsic) state of many fine-grained objects, keeping only the unique (extrinsic) state per use.

When (LLD): millions of objects with mostly identical data — glyphs in a text editor, tree sprites in a game, particles.

class GlyphFactory:
    _cache = {}
    @classmethod
    def get(cls, char):              # shared intrinsic state
        if char not in cls._cache:
            cls._cache[char] = Glyph(char)   # font, shape...
        return cls._cache[char]

class Glyph:
    def __init__(self, char): self.char = char
    def render(self, x, y):          # extrinsic state passed in
        print(self.char, "at", x, y)

a1 = GlyphFactory.get("a")
a2 = GlyphFactory.get("a")
assert a1 is a2     # one shared object for every 'a'

Pythonic idiom: @functools.lru_cache on a factory, string interning, or __slots__ to shrink per-instance memory.

Gotcha: shared flyweights must be immutable; if intrinsic state is mutated, every user is affected. Only optimize when memory is a proven bottleneck.

12. Proxy Structural

Intent: provide a surrogate that controls access to another object — for lazy loading, access control, caching, or logging.

When (LLD): virtual proxy (defer expensive init), protection proxy (auth checks), remote proxy (network stub), caching proxy.

class RealImage:
    def __init__(self, path):
        self.path = path
        print("loading", path)       # expensive
    def display(self): print("show", self.path)

class ImageProxy:                     # same interface
    def __init__(self, path): self.path, self._real = path, None
    def display(self):
        if self._real is None:        # lazy load on first use
            self._real = RealImage(self.path)
        self._real.display()

img = ImageProxy("big.png")           # nothing loaded yet
img.display()                          # now it loads + shows

Pythonic idiom: @property for lazy attributes, functools.cached_property, __getattr__ delegation, or descriptors for access control.

Gotcha: Proxy keeps the same interface and controls access; Decorator keeps the same interface and adds behavior; Adapter changes the interface. The intent is what separates them.

BEHAVIORAL · 11

Behavioral Patterns

How responsibilities are assigned and how objects communicate.

13. Chain of Responsibility Behavioral

Intent: pass a request along a chain of handlers; each either handles it or forwards to the next.

When (LLD): middleware stacks, approval workflows (team lead → manager → VP by amount), event/exception escalation, logging levels.

class Handler:
    def __init__(self, nxt=None): self.nxt = nxt
    def handle(self, amount):
        if self.nxt: return self.nxt.handle(amount)
        raise ValueError("unhandled")

class TeamLead(Handler):
    def handle(self, amount):
        if amount <= 1000: return f"TeamLead approved {amount}"
        return super().handle(amount)

class Manager(Handler):
    def handle(self, amount):
        if amount <= 10000: return f"Manager approved {amount}"
        return super().handle(amount)

chain = TeamLead(Manager())
print(chain.handle(5000))   # Manager approved 5000

Pythonic idiom: a list of predicate→handler callables you loop over, or composed middleware functions (the WSGI/ASGI model).

Gotcha: decide the contract for an unhandled request — return None, raise, or have a guaranteed default handler at the tail.

14. Command Behavioral

Intent: encapsulate a request as an object, letting you parameterize, queue, log, and undo operations.

When (LLD): undo/redo stacks, task queues, macro recording, transactional operations, remote/CLI command dispatch.

class Command(ABC):
    @abstractmethod
    def execute(self): ...
    @abstractmethod
    def undo(self): ...

class AddText(Command):
    def __init__(self, doc, text): self.doc, self.text = doc, text
    def execute(self): self.doc.append(self.text)
    def undo(self): self.doc.pop()

doc, history = [], []
cmd = AddText(doc, "hello"); cmd.execute(); history.append(cmd)
history.pop().undo()        # undo last command

Pythonic idiom: a callable or functools.partial(func, *args) is a command; store a list of them. Add undo only when you need reversibility.

Gotcha: undo must capture enough state to reverse precisely — pair with Memento when the operation isn't trivially invertible.

15. Interpreter Behavioral

Intent: given a language, define a representation for its grammar plus an interpreter that evaluates sentences in it.

When (LLD): small DSLs — search filters, rule engines, math/boolean expressions, feature-flag conditions.

class Expr(ABC):
    @abstractmethod
    def eval(self, ctx): ...

class Num(Expr):
    def __init__(self, v): self.v = v
    def eval(self, ctx): return self.v

class Add(Expr):
    def __init__(self, a, b): self.a, self.b = a, b
    def eval(self, ctx): return self.a.eval(ctx) + self.b.eval(ctx)

# (1 + 2) + 4
tree = Add(Add(Num(1), Num(2)), Num(4))
print(tree.eval({}))   # 7

Pythonic idiom: for real grammars use ast, lark, or pyparsing rather than hand-rolling node classes. Interpreter is mostly educational.

Gotcha: it scales poorly — each grammar rule is a class. Beyond a tiny grammar, switch to a real parser. Never eval() untrusted input.

16. Iterator Behavioral

Intent: provide sequential access to elements of a collection without exposing its underlying representation.

When (LLD): custom traversal order, lazy/streamed sequences, or hiding a complex internal structure (tree, graph, paginated API) behind a simple loop.

# Idiomatic: implement the iterator protocol
class CountDown:
    def __init__(self, n): self.n = n
    def __iter__(self): return self
    def __next__(self):
        if self.n <= 0: raise StopIteration
        self.n -= 1
        return self.n + 1

for x in CountDown(3): print(x)   # 3 2 1

# Even simpler: a generator
def count_down(n):
    while n > 0:
        yield n; n -= 1

Pythonic idiom: this is built into the language — __iter__/__next__ and especially generators (yield). You almost never write a separate iterator class.

Gotcha: an object that returns self from __iter__ is single-use; return a fresh iterator if you need to loop multiple times.

17. Mediator Behavioral

Intent: centralize complex communication between objects in a mediator, so they no longer refer to each other directly.

When (LLD): chat rooms (users talk via the room), air-traffic control, UI dialogs where widgets affect each other — turning a many-to-many mesh into a hub.

class ChatRoom:                       # mediator
    def __init__(self): self.users = []
    def join(self, u): self.users.append(u); u.room = self
    def broadcast(self, sender, msg):
        for u in self.users:
            if u is not sender: u.receive(msg)

class User:
    def __init__(self, name): self.name, self.room = name, None
    def send(self, msg): self.room.broadcast(self, msg)
    def receive(self, msg): print(self.name, "got:", msg)

room = ChatRoom(); a, b = User("A"), User("B")
room.join(a); room.join(b); a.send("hi")

Pythonic idiom: an event bus / pub-sub dict ({event: [callbacks]}) is a lightweight mediator. blinker signals are a ready-made one.

Gotcha: the mediator can swell into a god-object. Observer decouples one-to-many notification; Mediator coordinates many-to-many logic.

18. Memento Behavioral

Intent: capture and externalize an object's internal state so it can be restored later — without violating encapsulation.

When (LLD): save points / checkpoints, editor undo (paired with Command), transaction rollback, game state snapshots.

class Editor:
    def __init__(self): self.text = ""
    def save(self):                  # create memento
        return {"text": self.text}
    def restore(self, memento):      # restore from it
        self.text = memento["text"]

ed = Editor(); ed.text = "v1"
snapshot = ed.save()
ed.text = "v2"
ed.restore(snapshot)
print(ed.text)   # v1

Pythonic idiom: copy.deepcopy(self.__dict__), pickle, or dataclasses.replace capture state cheaply. The "caretaker" is often just a list of snapshots.

Gotcha: snapshots can be memory-heavy — store diffs or cap history. The caretaker must treat the memento as opaque (no peeking at internals).

19. Observer ← visualized above Behavioral

Intent: define a one-to-many dependency so that when one object changes state, all its dependents are notified automatically.

When (LLD): event systems, MVC (model → views), reactive UIs, pub-sub, stock tickers, the live demo at the top of this page.

class Subject:
    def __init__(self): self._observers = []
    def attach(self, obs): self._observers.append(obs)
    def notify(self, data):
        for obs in self._observers:
            obs.update(data)          # push to each

class Logger:
    def update(self, data): print("log:", data)

s = Subject(); s.attach(Logger())
s.notify("temp=72")

Pythonic idiom: a list of callbacks (self._subscribers.append(fn)), the blinker library, or property setters that fire listeners. Observers need only be callables.

Gotcha: leaked subscriptions cause memory leaks (keep detach() / weak refs); notification order isn't guaranteed; an observer raising can break the loop — isolate failures.

20. State Behavioral

Intent: let an object alter its behavior when its internal state changes — it appears to change its class.

When (LLD): finite state machines — order lifecycle (draft → paid → shipped), TCP connection, vending machine, document workflow. Replaces sprawling if state == ... blocks.

class State(ABC):
    @abstractmethod
    def next(self, order): ...

class Draft(State):
    def next(self, order): order.state = Paid(); print("-> Paid")
class Paid(State):
    def next(self, order): order.state = Shipped(); print("-> Shipped")
class Shipped(State):
    def next(self, order): print("already shipped")

class Order:
    def __init__(self): self.state = Draft()
    def advance(self): self.state.next(self)

o = Order(); o.advance(); o.advance()   # -> Paid -> Shipped

Pythonic idiom: a transition table ({(state, event): new_state}), an enum.Enum + dict, or libraries like transitions for richer FSMs.

Gotcha: State and Strategy share the structure; the difference is intent — State objects know about and trigger transitions to each other; Strategy objects are independent and chosen by the client.

21. Strategy Behavioral

Intent: define a family of interchangeable algorithms, encapsulate each, and make them swappable at runtime.

When (LLD): pluggable sort/compression/pricing/routing algorithms; payment methods; A/B-swappable behavior chosen by config or context.

# Pythonic Strategy: pass a function
def by_price(items): return sorted(items, key=lambda x: x["price"])
def by_name(items):  return sorted(items, key=lambda x: x["name"])

class Catalog:
    def __init__(self, strategy): self.strategy = strategy
    def list(self, items): return self.strategy(items)

data = [{"name": "b", "price": 2}, {"name": "a", "price": 9}]
print(Catalog(by_price).list(data))   # swap strategy freely

Pythonic idiom: the textbook Strategy interface collapses to a first-class function or any callable. The key= argument to sorted() is Strategy in the stdlib.

Gotcha: don't build a 4-class hierarchy when a function suffices. Use classes only when strategies carry their own state or config.

22. Template Method Behavioral

Intent: define the skeleton of an algorithm in a base method, deferring specific steps to subclasses.

When (LLD): fixed pipelines with pluggable steps — ETL (extract/transform/load), test setup/run/teardown, data parsers sharing one flow.

class Pipeline(ABC):
    def run(self):                   # the template (fixed skeleton)
        data = self.extract()
        data = self.transform(data)
        self.load(data)
    @abstractmethod
    def extract(self): ...
    @abstractmethod
    def transform(self, d): ...
    def load(self, d): print("loaded", d)   # default hook

class CSVPipeline(Pipeline):
    def extract(self): return [1, 2, 3]
    def transform(self, d): return [x * 10 for x in d]

CSVPipeline().run()   # loaded [10, 20, 30]

Pythonic idiom: pass the varying steps as functions (that's Strategy) instead of subclassing. Template Method = inheritance; Strategy = composition — prefer the latter unless the skeleton truly belongs in a base class.

Gotcha: deep inheritance and the "fragile base class" problem — changing the template can silently break every subclass.

23. Visitor Behavioral

Intent: represent an operation to be performed on elements of an object structure, letting you add new operations without modifying those classes.

When (LLD): AST traversal (type-check, compile, pretty-print), file-tree operations, exporting a model to many formats — a stable class hierarchy but a growing set of operations.

class Circle:
    def __init__(self, r): self.r = r
    def accept(self, v): return v.visit_circle(self)
class Square:
    def __init__(self, s): self.s = s
    def accept(self, v): return v.visit_square(self)

class AreaVisitor:                    # new op, no shape edits
    def visit_circle(self, c): return 3.14 * c.r ** 2
    def visit_square(self, s): return s.s ** 2

shapes = [Circle(1), Square(2)]
print(sum(sh.accept(AreaVisitor()) for sh in shapes))

Pythonic idiom: functools.singledispatch dispatches on type without the double-dispatch boilerplate — register one function per node type.

Gotcha: Visitor makes adding operations easy but adding a new element type hard (touch every visitor) — the "expression problem". Choose based on which axis changes more.

SOLID Foundations & When NOT to Use Patterns

Single Responsibility — one reason to change per class.

Open/Closed — open to extension, closed to modification. (Strategy, Decorator)

Liskov Substitution — subtypes must be usable as their base.

Interface Segregation — many small interfaces beat one fat one.

Dependency Inversion — depend on abstractions, not concretions. (Factory, Bridge)

Pattern overuse is a smell

Don't reach for a pattern until the duplication or rigidity actually hurts. In Python especially, a function, a dict dispatch table, or a module-level object often beats a five-class pattern. Patterns are vocabulary for real forces — not a checklist to satisfy.
