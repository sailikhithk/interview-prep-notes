1. Real-World Reference & Case Study

Production Anchor: YouTube's Vitess scaling

YouTube created Vitess to scale their backend MySQL database infrastructure horizontally. Faced with massive read/write scale requests, they wrapped thousands of MySQL instances with Vitess routing proxies. By managing shard maps inside etcd, Vitess allowed YouTube to divide database shards live, reducing hardware costs and eliminating cross-shard transaction bottlenecks.

2. Requirements & Scope Clarification

Functional

  • Horizontal Partitioning: Shard high-volume tables across independent database engines.
  • Transparent Routing: Match incoming writes to target shards based on the partition key.
  • Live Migration: Split and move shard ranges online when storage capacities exceed bounds.

Non-Functional

  • Extreme Scalability: Scaling write throughput linearly up to 100K+ QPS.
  • High Availability: Primary-replica replication setups per shard with rapid failovers.
  • Low overhead: Query lookup proxies must execute routing calculations in under 2ms.

Out of Scope

  • Custom database kernel modifications or disk controller buffer tuning.
  • Replicating global locks across the entire cluster natively (AP-favored).
3. Scale & Capacity Estimation (Back-of-the-Envelope)
Storage Capacity Ingestion Rate
100,000 writes/sec with average size of 500 bytes: $$ Ingestion\_Rate = 100,000 \times 500\text{ bytes} = 50\text{ MB/s} $$ $$ Ingestion\_Daily = 50\text{ MB/s} \times 86,400\text{s} \approx 4.32\text{ TB/day} $$
5-Year Storage Size
Compiling storage requirements over 5 years: $$ Storage_{5Y} = 4.32\text{ TB/day} \times 365 \times 5 \approx 7.884\text{ PB} $$ Assuming physical limits of 10 TB storage per node: $$ Shards_{Count} = \frac{7,884\text{ TB}}{10\text{ TB}} \approx 789\text{ Shard Databases} $$
Index Cache RAM Sizing

Caching the primary index segments of the user orders table (10% of overall data size):

$$ Index\_Storage = 7.884\text{ PB} \times 0.05 = 394\text{ TB} $$ $$ Caching\_Index = 394\text{ TB} \times 0.10 \approx 39.4\text{ TB of RAM} $$ Using Redis cluster hosting 128 GB memory instances: $$ Cache\_Nodes = \frac{39.4\text{ TB}}{128\text{ GB}} \approx 308\text{ Cache Nodes} $$

5. Read & Write Data Flow Traces
Read Path Flow (Blue Path)
  1. Client sends SELECT * FROM orders WHERE user_id = 'user_82739' query.
  2. Proxy intercepts request and queries its local memory TreeMap cache to determine target shard.
  3. Proxy resolves destination to Shard Node 03 master (or replica for eventual consistency).
  4. Proxy connects, fetches matching rows, and returns array to Client application.
Write Path Flow (Orange Path)
  1. Client issues a SQL Insert payload.
  2. Proxy routes request to Shard Node 03 (Master) based on sharding key user_id hash range.
  3. Shard Node 03 commits transaction.
  4. Shard Node 03 triggers background asynchronous log replication to replicas.

8. Distributed Coordination & State Management
To implement horizontal sharding transparently, Vitess uses an etcd cluster as the global Lockserver/Metadata registry. API routing maps and shard ranges are managed inside etcd. When a shard splits, updates sync to proxy memories asynchronously, protecting continuous execution.
9. Detailed Partitioning & Database Scaling (Sharding)
By selecting a high-cardinality shard key (like user_id), we distribute records across multiple databases uniformly, ensuring SQL read and write execution scales horizontally.
10. Consistent Hashing & Ring Mechanics
Consistent hashing mapping is utilized to assign data keys dynamically to available shard database nodes, ensuring range expansions or removals require minimal row relocation.
11. Caching Topology & Invalidation Strategy
Metadata Routing Cache

Local Proxy Caching: Routing proxies cache the entire shard range directory mapping locally in RAM. This ensures key routing analysis requires zero network hops to etcd configuration directories. Watchers invalidate routing entries instantly on resharding swaps.

Data Cache Placement

Cache-Aside Redis: Placed directly in front of each physical master database shard to handle read spikes. Redis nodes are partitioned using modular hashing based on the same key structure, keeping cache nodes localized to shards.

12. Asynchronous Processing & Message Queues
Event-driven pipelines decouple write tasks. For example, during shard splitting, updates trigger Kafka messaging topics where background sync workers verify row checksum integrity before promoting target split nodes.

Distributed Snowflake ID Generator (Java Implementation) 
public class SnowflakeIdGenerator {
    private final long workerId;
    private long sequence = 0L;
    private long lastTimestamp = -1L;

    public SnowflakeIdGenerator(long workerId) {
        this.workerId = workerId;
    }

    public synchronized long nextId() {
        long timestamp = System.currentTimeMillis();
        if (timestamp == lastTimestamp) {
            sequence = (sequence + 1) & 4095;
            if (sequence == 0) {
                timestamp = waitNextMillis(lastTimestamp);
            }
        } else {
            sequence = 0L;
        }
        lastTimestamp = timestamp;
        return ((timestamp - 1577836800000L) << 22) | (workerId << 12) | sequence;
    }

    private long waitNextMillis(long lastTimestamp) {
        long timestamp = System.currentTimeMillis();
        while (timestamp <= lastTimestamp) {
            timestamp = System.currentTimeMillis();
        }
        return timestamp;
    }
}
Distributed Snowflake ID Generator (Python Implementation) 
import time
import threading

class SnowflakeIdGenerator:
    def __init__(self, worker_id: int):
        self.worker_id = worker_id & 1023
        self.sequence = 0
        self.last_timestamp = -1
        self.lock = threading.Lock()

    def next_id(self) -> int:
        with self.lock:
            timestamp = int(time.time() * 1000)
            if timestamp < self.last_timestamp:
                raise Exception("Clock moved backwards. Refusing to generate id.")
            
            if timestamp == self.last_timestamp:
                self.sequence = (self.sequence + 1) & 4095
                if self.sequence == 0:
                    timestamp = self._wait_next_millis(self.last_timestamp)
            else:
                self.sequence = 0
                
            self.last_timestamp = timestamp
            epoch_offset = 1577836800000
            return ((timestamp - epoch_offset) << 22) | (self.worker_id << 12) | self.sequence

    def _wait_next_millis(self, last_timestamp: int) -> int:
        timestamp = int(time.time() * 1000)
        while timestamp <= last_timestamp:
            timestamp = int(time.time() * 1000)
        return timestamp
14. Resiliency & Failure Mitigations (Scenario Cards)

Scenario 1: Single Shard Outage

Impact: If Shard Node 03 crashes, 1/N of the overall users experience total write and read failures, causing cascading errors across user dashboards.

Mitigation: Deploy automated primary-replica failovers. The sharding proxy redirects read traffic immediately to replicas and blocks writes while replica promotes to master via consensus elections.

Scenario 2: Shard Key Hotspotting (Viral User Event)

Impact: A viral user gets millions of concurrent writes, flooding a single physical shard node with traffic, saturating disk IOPS, and locking threads.

Mitigation: Create dynamic sub-shards for high-traffic keys. Append random suffixes (e.g. `viral_user#1`, `viral_user#2`) to partition keys dynamically to spread load across multiple nodes, combined with application-level write-buffering.

15. Security, Auth & Data Protection
All sharded database node communication is secured via mTLS. App proxy levels enforce rate limits using Token Bucket algorithms to protect shards from DDoS attacks.

20. Advanced Concept Accordions
Q: How do you generate unique, sequential IDs across multiple shards without a single bottleneck? ▾

We avoid database auto-increment fields since shards do not coordinate. Instead, we use Snowflake-style distributed ID generators. These create a 64-bit integer composed of a timestamp prefix, machine node identifier, and a local sequence number. This guarantees globally unique, time-sortable keys generated locally on application nodes in O(1) time.

Q: How do you handle cross-shard transactions? ▾

We avoid standard Two-Phase Commit (2PC) protocols because they block threads and create bottlenecks under high traffic. Instead, we design transactions around the Saga Pattern. Sagas break transactions into local shard tasks executed sequentially. If one shard task fails, compensating transactions are triggered backwards to roll back the previous states, favoring eventual consistency (AP) over strict immediate consistency (CP).

Q: What is the difference between vertical sharding and horizontal sharding? ▾

Vertical sharding splits columns of a table into separate databases (e.g. separating user login details from profile images to optimize disk access). Horizontal sharding splits rows of the same table across separate instances based on range or hash criteria. This study workbook targets horizontal sharding.

Q: What is directory-based sharding and what are its performance trade-offs? ▾

Directory sharding uses an external lookup database (e.g. Redis or etcd) to map shard keys to physical shard nodes individually. It provides absolute mapping flexibility, but introduces a query hop to resolve coordinates, making lookup cache maintenance and availability critical path bottlenecks.

Q: How do Vitess and Citus differ in their sharding execution? ▾

Vitess functions as a proxy coordinator tier in front of standard MySQL database instances, routing SQL query trees transparently. Citus is a PostgreSQL extension that integrates sharding directly inside the database engines, using Postgres query planners to distribute tables and coordinate maps locally.
