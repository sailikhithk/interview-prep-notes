Sliding Window Maximum

Given an array of integers and a sliding window of size k, return the maximum element inside the window at each position as it moves from left to right.

Array Sliding Window Monotonic Deque
TimeO(N)
SpaceO(K)

1. Problem Statement

You are given an array of integers nums, there is a sliding window of size k which is moving from the very left of the array to the very right. You can only see the k numbers in the window. Each time the sliding window moves right by one position.

Return the max sliding window.

Example 1: 
Input: nums = [1,3,-1,-3,5,3,6,7], k = 3
Output: [3,3,5,5,6,7]
Explanation: 
Window position                Max
------------------------      -----
[1  3  -1] -3  5  3  6  7       3
 1 [3  -1  -3] 5  3  6  7       3
 1  3 [-1  -3  5] 3  6  7       5
 1  3  -1 [-3  5  3] 6  7       5
 1  3  -1  -3 [5  3  6] 7       6
 1  3  -1  -3  5 [3  6  7]      7
Constraints: 1 ≤ nums.length ≤ 105 · 1 ≤ k ≤ nums.length · -104 ≤ nums[i] ≤ 104
2. Pattern Recognition
Pattern Monotonic Deque + Sliding Window

When you need to keep track of extreme statistics (min/max) within a dynamically shifting range, and scanning the range repeatedly is too slow.

How to spot it:

The problem involves a sliding window range of width k.

You need to perform extreme value lookups (min or max) in O(1) time at each window shift.

New elements render older, smaller/larger elements obsolete (obsolescence invariant).

3. Pattern Blueprint Template

Use this blueprint for any sliding window extreme lookup problem:

1

Initialize a double-ended queue (deque) to store indices.

2

Evict out-of-bounds indices from the front of the deque: i - deque[0] >= k.

3

Pop indices from the back of the deque while the current element is larger than (or smaller than, for minimums) the elements represented by the back indices.

4

Append the current index i to the back of the deque.

5

Once the window reaches size k, the element at the index stored in deque[0] is the extreme value.

4. How to Think About It Key Section
1

"A naive loop checks k elements at each of the N window shifts. That's O(N * K). Can we avoid looking backwards?"

As the window shifts, only one element enters and one element leaves. We shouldn't scan the whole window again. We need a way to maintain the maximum candidate incrementally.

→ Transform: Instead of recalculating, maintain a running list of maximum candidates.

2

"If a new element is larger than an older element inside the window, does that older element matter anymore?"

No. Because the new element is larger AND it will stay in the sliding window longer than the older, smaller element. The older, smaller element can NEVER be the maximum in any future window. We can prune it immediately.

→ Evict smaller elements from our tracking structures when a larger element joins.

3

"We need to access the max element at the front, and prune obsolete elements from the back. What data structure supports this?"

A Double-Ended Queue (Deque). By keeping indices in the deque, we can easily check if the front element is out-of-bounds, while popping smaller values from the back to maintain a monotonic descending order.

→ Result: Amortized O(1) operations per element, yielding O(N) total time.

5. Real-World Analogy

The Career Promotion Ladder

Imagine a sliding window as a company's promotion window representing the last k years of hires. We want to know who the most qualified person (highest value) is at any time.

When a highly qualified newcomer (larger value) joins the company, anyone who joined before them (older indices) but has less qualifications (smaller values) is immediately overshadowed. They will never be promoted over the newcomer. Thus, the manager prunes them from the promotion queue.

Additionally, employees who reach retirement limits (indices exceeding the sliding window bounds) are removed from the front of the queue. The top candidate eligible for promotion is always at the front of the queue.

The Deque stores index candidates in strictly decreasing order of qualifications.

6. Build Your Intuition

"Why does sorting or priority queues fail to reach linear O(N) time?"

A Priority Queue (Heap) would take O(log K) to push and pop elements on each shift, leading to O(N log K). We avoid this log factor by recognizing the "monotonic obsolescence" invariant—meaning we can discard intermediate values immediately without sorting them, because they are guaranteed never to matter.

7. How It Works (Step by Step)

Example: nums = [1, 3, -1], k = 3

1

Initialize empty deque: []

2

Process i=0, x=1. Deque empty. Store index [0].

3

Process i=1, x=3. Since 3 >= nums[0] (1), pop index 0. Store index [1].

4

Process i=2, x=-1. Since -1 < nums[1] (3), append index 2. Deque: [1, 2]. Output deq[0] element = 3.

10. Complexity Deep Dive
Time ComplexityO(N)

Each index element is appended to the deque exactly once and popped from the deque at most once. The amortized complexity per operation is strictly O(1).

Space ComplexityO(K)

The double-ended queue stores at most k index elements, since indices that have shifted outside of the active sliding window bounds are immediately evicted.

11. From Brute Force to Optimal
Naive O(N * K)

Scan the entire sliding window of size k at each index position to extract the maximum element.

Optimize
Optimal O(N)

Maintain a running monotonic descending deque of maximum candidates. Evict indices out-of-bounds at the head and smaller elements at the tail.

Concrete speedup metric:

With N = 100,000 and K = 10,000, naive requires 1 Billion checks. Monotonic Deque does it in ~200,000 operations, resulting in a 5,000x speedup.

12. Edge Cases & Gotchas

Window size k = 1

Handle: Output is a copy of input array. Correctly processed immediately at index 0.

Strictly increasing array (e.g. [1, 2, 3, 4])

Handle: Deque size stays at 1 because every incoming larger element pops all tail elements.

Strictly decreasing array (e.g. [4, 3, 2, 1])

Handle: No tail pops take place; deque grows to size k before front evictions occur.

13. Flashcards 5 cards
Q: Why use indices in the deque instead of values?

Indices allow us to perform out-of-bounds checks to evict elements that have aged out of the sliding window range. If we stored values, we wouldn't know when they expired.

Q: What is the monotonic property maintained here?

Monotonic descending order: the element at the head of the deque is always the largest candidate, followed by progressively smaller candidates.

Q: What is the worst-case space complexity of the deque?

O(K) where k is the size of the sliding window. This occurs in a strictly decreasing array where no tail pops happen.

Q: Can this be solved with a Heap?

Yes, a Max-Heap storing (value, index) pairs works in O(N log N) by discarding expired heap root entries on shifts. The deque optimizes this to O(N) linear time.

Q: What is the crossover threshold where naive beats deque?

For window sizes k ≤ 4, the overhead of heap allocations and double-ended node pointer adjustments in the deque makes brute-force loops faster due to stack cache locality.

15. Common Mistakes

Evicting from the front using value boundaries instead of index age bounds (deq[0] < i - k + 1).

Using standard array shift operations (shift()) in JavaScript inside loops, creating hidden O(N) costs leading to quadratic O(N²) execution.

Forgetting to start adding elements to the result array only after the loop index reaches k - 1.

class Solution:
    def maxSlidingWindow(self, nums: List[int], k: int) -> List[int]:
        deq = deque()
        result = []
        for i, x in enumerate(nums):
            if deq and deq[0] < i - k + 1:
                deq.popleft()
            while deq and nums[deq[-1]] <= x:
                deq.pop()
            deq.append(i)
            if i ≥ k - 1:
                result.append(nums[deq[0]])
        return result
Java 
class Solution {
    public int[] maxSlidingWindow(int[] nums, int k) {
        Deque<Integer> deq = new ArrayDeque<>();
        int[] result = new int[nums.length - k + 1];
        int idx = 0;
        for (int i = 0; i < nums.length; i++) {
            if (!deq.isEmpty() && deq.peekFirst() < i - k + 1) {
                deq.pollFirst();
            }
            while (!deq.isEmpty() && nums[deq.peekLast()] <= nums[i]) {
                deq.pollLast();
            }
            deq.offerLast(i);
            if (i ≥ k - 1) {
                result[idx++] = nums[deq.peekFirst()];
            }
        }
        return result;
    }
}

class Solution {

    vector<int> maxSlidingWindow(vector<int>& nums, int k) {
        deque<int> deq;
        vector<int> result;

            if (!deq.empty() && deq.front() < i - k + 1) {
                deq.pop_front();
            }
            while (!deq.empty() && nums[deq.back()] <= nums[i]) {
                deq.pop_back();

function maxSlidingWindow(nums: number[], k: number): number[] {
    const deq: number[] = [];
    const result: number[] = [];
    for (let i = 0; i < nums.length; i++) {
        if (deq.length > 0 && deq[0] < i - k + 1) {
            deq.shift();
        }
        while (deq.length > 0 && nums[deq[deq.length - 1]] <= nums[i]) {
            deq.pop();
        }
        deq.push(i);
        if (i ≥ k - 1) {
            result.push(nums[deq[0]]);
        }
    }
    return result;
}
21. Follow-Up Questions Senior Interviews
What if the input size N is infinite?

The monotonic deque operates as an event-driven stream aggregator. Since it only stores indices within the active window length k, the space complexity remains bounded at O(K) even over infinite input streams, processing each element in O(1) time.

What if we cannot fit the window of size k in memory?

If k is extremely large (e.g. trillions of elements), partition the array into block segments on disk. Run a local monotonic deque on each segment, and consolidate segment boundary max outputs sequentially using a disk-backed merge protocol.
